## 0. 프로젝트 컨텍스트 (요약)

DGCC는 rope(DLO) 조작에서 contact point 선택을 위한 critic 표현 연구다. P0는 **RL 이전 단계**로, (1) 시뮬레이터 2종을 실제 구동해 비교하고, (2) rope 상태·행동 인터페이스와 transition 로깅을 만들고, (3) δm(변형 반응) 파이프라인을 구현하고, (4) 두 파일럿 게이트(G1 stiffness 유효성, G2 length-goal 정의)의 **측정 자료를 생성**하는 단계다. 게이트의 **판정은 사람이 한다** — gjc는 측정과 보고까지만 수행한다.

## 5. 공통 env 인터페이스 (M0에서 골격 정의, M1/M3에서 구현)

두 시뮬레이터는 동일한 추상 인터페이스의 adapter로 구현한다. 단위는 m, 좌표는 world(task) frame.

```python
@dataclass
class RopeParams:
    length_m: float          # 물리 길이
    n_segments: int          # 시뮬 내부 세그먼트 수 (기본 50)
    bend_stiffness: float    # sim별 단위로 매핑 (adapter 책임)
    twist_stiffness: float
    friction: float          # (mu_s 기준; adapter가 sim별로 매핑)
    radius: float = 0.005

class DLOEnvBase(ABC):
    K: int = 32              # 재표본 노드 수 (외부 인터페이스 기준)
    def reset(self, params: RopeParams, init_shape: str, seed: int) -> dict: ...
    def get_centerline_raw(self) -> np.ndarray        # (N_sim, 3)
    def get_centerline(self) -> np.ndarray            # (32, 3) arc-length 재표본
    def step_primitive(self, p: int, delta: np.ndarray, lift: str) -> dict:
        # grasp(node p) → move(delta∈R³, ‖delta‖≤0.15, lift∈{"low","high"}) → release → settle
        # 반환: {"X_before": (32,3), "X_after": (32,3), "grasp_success": bool,
        #        "settle_steps": int, "info": {...}}
    def settle(self, vel_threshold: float = 1e-3, max_steps: int = 5000) -> bool
```

lift 높이: low = 0.02 m, high = 0.15 m. settle = 최대 노드 속도(또는 qvel 최대 절대값)가 임계 아래로 떨어질 때까지 step (quasi-static 보장).

## 6. 시뮬레이터 기술 사실 (조사 완료 — 구현 시 이 사실을 기준으로)

### 6.1 DLO-Lab (후보 A)

- Repo: `https://github.com/UMass-Embodied-AGI/DLO-Lab` (Apache-2.0, Genesis fork, `genesis-world==1.0.0`, 백엔드 `quadrants==0.8.0` — cp310–cp313 manylinux_2_28 wheel 존재, Ubuntu 22.04 OK). Python 3.12 권장, `torch`는 공식 지침대로 별도 설치, extras `.[dlo-lab]`.
- Rope 생성: `gs.morphs.ParameterizedRod(type="rod", n_vertices≈50, interval, radius, rest_state="straight")` — 길이 ≈ `(n_vertices-1)*interval`.
- Centerline: `rod_entity.sample_centerline(n_samples=32)` → `(n_envs, 32, 3)` (arc-length 균등 재표본 내장).
- 파라미터 setter (런타임, per-env): `set_bending_stiffness`, `set_twisting_stiffness`, `set_stretching_stiffness`, `set_mu_s`, `set_mu_k`, `set_segment_mass`, `set_segment_radius`. Material: `gs.materials.ROD.Base(K, E, G, static_friction, kinetic_friction, plastic_yield, plastic_creep, ...)`.
- Grasp: `rod_entity.attach_to_rigid_link(rigid_link, verts_ids)` / `detach_from_rigid_link(verts_ids)` (per-env 변형 존재) — 노드 인덱스 지정 grasp가 내장. 이동은 attach된 rigid link를 kinematic하게 이동.
- Headless/병렬: `gs.init(backend=gs.gpu)` + `show_viewer=False`, `scene.build(n_envs=N)` GPU-batched.
- 주의: 공개 5주차, CI/테스트 없음. asset은 SharePoint 링크 — 다운로드 후 로컬 보관(커밋 금지). 벤치마크 env 클래스(`experiments/envs/…`, mushroom-rl 의존)는 사용하지 말고 **저수준 `rod_entity` API로 직접 adapter를 작성**한다.

### 6.2 MuJoCo cable (후보 B)

- `pip install mujoco` (3.5+ 권장). Rope = `<composite type="cable">` + `mujoco.elasticity.cable` 플러그인. **현행 MuJoCo에서 composite는 cable만 지원** (rope/grid 등 legacy 타입 사용 금지).
- MJCF 골격 (검증된 예시 기반):

```xml
<extension><plugin plugin="mujoco.elasticity.cable"/></extension>
<size memory="4M"/>
<worldbody>
  <composite type="cable" curve="s" count="50 1 1" size="1.0" initial="free" prefix="rope">
    <plugin plugin="mujoco.elasticity.cable">
      <config key="twist" value="1e7"/><config key="bend" value="4e6"/>
    </plugin>
    <joint kind="main" damping="0.015" armature="0.01"/>
    <geom type="capsule" size="0.005" condim="3" friction="0.3"/>
  </composite>
  <body name="grip_mocap" mocap="true"><geom type="sphere" size="0.005" contype="0" conaffinity="0"/></body>
</worldbody>
<equality><weld name="grasp" body1="ropeB25" body2="grip_mocap" active="false" torquescale="0"/></equality>
```

- 길이/세그먼트 변경 = MJCF 문자열 재생성 (`MjModel.from_xml_string`). 생성 body 이름은 `ropeB0..ropeB{N-1}` (`B_first/B_last` 아님 — 하드코딩 금지, `mj_name2id`로 열거).
- Centerline: `data.xpos[body_ids]` → (N,3); 32점 arc-length 재표본은 우리 코드(`phi/resample.py`)로.
- Grasp: 노드별 weld를 미리 두거나 grasp 시점에 대상 body로 weld 지정 → **`data.mocap_pos`를 노드 현재 위치로 세팅한 뒤** `data.eq_active[weld_id]=1`, 이동은 mocap_pos 보간, release는 `=0` 후 settle. (mocap을 노드 위치에 맞추지 않고 weld를 켜면 rope가 튕긴다 — 반드시 선 세팅.)
- 안정화: `<option integrator="implicitfast" timestep="0.001"/>`, solver iterations 상향, 인접 세그먼트 contact exclude. stiffness는 modulus×geometry(반지름⁴ 스케일)라 물리 단위 직결 아님 — 상대 배율(×0.5/×2)로 다룬다.
- **CPU 전용** (MJX는 cable 플러그인 미지원). 병렬화는 이번 단계에서 불필요 — 단일 프로세스로 충분.

## 7. δm 파이프라인 스펙 (M4 구현 대상)

```
resample(X_raw: (N,3)) -> (32,3)          # 정규화 호길이 균등 32점, 선형 보간
Phi_DCT(X: (32,3)) -> (24,)               # 축별(x,y,z) DCT-II, 저차 M=8 mode
                                          # mode 0 = centroid(축별 평균) — 분리 취급
delta_m = Phi(X_after) - Phi(X_before)
normalize(delta_m, stats) -> (24,)        # 채널별 std 정규화, mode-0 채널 3개는 제외
```

Φ 불변성 요건: 동일한 물리 형상(해석적으로 정의된 곡선, 예: 반원, S-curve)을 N=25/50/100으로 이산화해 각각 Φ를 계산했을 때, mode≥1 채널의 상대 오차가 2% 미만이어야 한다 (`tests/test_phi_invariance.py`).

## 8. 이원 goal·거리 스펙 (M5 구현 대상)

```
goal = {shape_template: (32,3) 정규화 호길이 위 형상 (scale-normalized, centroid 제거),
        anchor: (3,) 절대 위치 목표 (rope centroid 또는 한 끝점 — config로 선택)}
c_g  = [Phi_shape(G) - Phi_shape(X)  (mode≥1, 21ch),  anchor(G) - anchor(X)  (3ch)]
D(X, G) = 길이 정규화 bidirectional Chamfer distance (점 개수 무관, 길이로 나눔)
성공 판정(잠정): D < 0.05 * length_m   # M7에서 사람이 확정
```

G2 정량 검증(게이트) — 성분 분해형 (2026-07-03 Decision으로 개정, 재시도 1회 소진):
  (a) anchor 성분: Spearman ρ(ΔD, Δ‖c_g_anchor‖) ≥ 0.9
      D = 절대 좌표 correspondence L2 (아래 정의), 기존 무작위 goal 표본, primary population
  (b) shape 성분: Spearman ρ(ΔD_shape, Δ‖c_g_shape‖) ≥ 0.9
      D_shape = centroid 정렬 + 길이 정규화 correspondence L2, 동일 population
  통과 = (a) AND (b). shape 성분 재미달 시 stopping rule — 추가 재시도 없이 human 재설계 결정 대기.
correspondence L2 정의 (K=32 arc-length 재표본 canonical 경로 사용):
  D(X,G) = min over orientation flip of (1/L)·sqrt((1/K)·Σ_k ‖X̃_k − G̃_k‖²)
  D_shape(X,G) = 동일하되 X̃, G̃ 각각 centroid 제거 후 계산
  orientation flip: rope 양끝 대칭성 — k ↔ K+1−k 대응 중 작은 값 채택
orientation canonicalization 규약 (2026-07-03 M5R2 Case A로 편입 — 버그 수정, 파라미터 변경 아님):
  per transition, choose exactly one flip decision against the goal using X_before, selecting the orientation that minimizes the mode-1..7 residual versus the goal's shape Phi
  apply that fixed decision identically to X_before and X_after in both c_g_shape and D_shape computations for the shape component
  component (a) absolute D keeps the existing correspondence_l2 min-flip unchanged
